# Технический документ: Telegram-бот записи к тату-мастеру

## 1. Актуальное состояние

Проект — aiogram Telegram-бот для записи к тату-мастеру/студии.

В актуальной версии реализовано:

- регистрация пользователя через `/start`;
- отдельные клиентское и админское меню;
- каталог эскизов;
- выбор стиля;
- выбор эскиза;
- карточка эскиза;
- клиентская кнопка `Запись`;
- создание заявки из каталожного эскиза;
- создание заявки с фото клиента;
- создание заявки без эскиза;
- клиентские `Мои заявки`;
- клиентские `Мои соцсети`;
- клиентский view-only календарь;
- админский календарь;
- админские заявки;
- админское рабочее время;
- админское управление эскизами;
- напоминания за день до подтверждённой записи;
- webhook mode по умолчанию.

`main` — ветка для Railway/webhook deployment без Tailscale/Funnel.
Ветка `tailscale-funnel` хранит альтернативный Docker Compose deployment с
Tailscale Funnel.

## 2. Клиентская логика записи

Клиент начинает запись через кнопку:

```text
Запись
```

После нажатия бот показывает варианты:

```text
Мой эскиз
Выбрать эскиз
Чат с мастером
Заявка без эскиза
Назад
Главное меню
```

## 3. Логика каталога

Каталог работает через FSM и открывается из меню `Запись` через кнопку
`Выбрать эскиз`.

Пользовательский путь:

```text
Запись
→ Выбрать эскиз
→ выбор стиля
→ выбор эскиза
→ карточка эскиза
```

После открытия карточки эскиза пользователь видит действия:

```text
Создать заявку
Чат с мастером
Назад
Главное меню
```

Отдельной кнопки `Оставить комментарий` в карточке эскиза нет. Комментарий
собирается позже внутри сценария создания заявки.

## 4. Главный пользовательский сценарий заявки

### Вариант 1. Заявка из каталожного эскиза

Пользователь открывает `Запись`, нажимает `Выбрать эскиз`, выбирает стиль и
конкретный эскиз.

После этого бот показывает карточку эскиза. Кнопка `Создать заявку` начинает
сценарий записи.

Бот должен сохранить в FSM:

```text
sketch_id
```

Если `sketch_id` отсутствует, бот должен ответить:

```text
Сначала выберите эскиз из каталога.
```

### Вариант 2. Заявка с фото клиента

Пользователь открывает `Запись`, нажимает `Мой эскиз`, затем `Пришлите ваш
эскиз`.

Бот принимает только Telegram photo, сохраняет `client_sketch_photo_file_id` в
FSM и создаёт заявку с:

```text
request_type = custom_sketch
sketch_id = null
цена = договорная
```

Админ видит фото клиента при просмотре заявки. `file_id` не должен логироваться.

### Вариант 3. Заявка без эскиза

Пользователь открывает `Запись` и нажимает `Заявка без эскиза`.

Заявка создаётся с:

```text
request_type = no_sketch
sketch_id = null
```

### Общие шаги заявки

### Шаг 1. Пользователь выбирает дату

Бот показывает календарь.

Календарь должен показывать только дни месяца.

Важно:

- в календаре не показываем временные отрезки;
- временные слоты показываем только после выбора конкретного дня;
- недоступные дни нельзя выбрать.

### Шаг 2. Пользователь выбирает время

После выбора даты бот показывает список доступных временных слотов.

Пример:

```text
Выберите время:

10:00
12:00
14:00
16:00
18:00
```

Слот не показывается, если он недоступен.

### Шаг 3. Пользователь оставляет комментарий

Комментарий можно сделать необязательным.

Бот спрашивает:

```text
Оставьте комментарий к заявке или нажмите «Пропустить».
```

Пример комментария:

```text
Хочу сделать на предплечье, размер примерно 10 см.
```

### Шаг 4. Пользователь подтверждает создание заявки

Бот показывает сводку:

```text
Проверьте заявку:

Эскиз: Череп с цветами
Дата: 15.07.2026
Время: 14:00
Комментарий: Хочу сделать на предплечье

Статус после создания: ждёт подтверждения
```

Кнопки:

```text
Создать заявку
Изменить дату
Изменить время
Изменить комментарий
Отмена
```

### Шаг 5. Заявка создаётся

После нажатия **Создать заявку** в базе создаётся заявка со статусом:

```text
pending
```

Пользователь получает сообщение:

```text
Заявка создана и ожидает подтверждения.

После разговора с мастером админ подтвердит или отклонит заявку.
```

## 5. Статусы заявок

У заявки должны быть статусы:

```text
pending
confirmed
rejected
cancelled
```

### `pending`

Заявка создана, но админ ещё не подтвердил её.

Показываем пользователю:

```text
Ждёт подтверждения
```

### `confirmed`

Админ подтвердил заявку.

Показываем пользователю:

```text
Подтверждена
```

### `rejected`

Админ отклонил заявку.

Показываем пользователю:

```text
Отклонена
```

### `cancelled`

Заявка отменена.

Можно добавить позже, но статус лучше заложить сразу.

## 6. Важное правило по слотам

Слот считается занятым, если есть заявка со статусом:

```text
pending
confirmed
```

Это нужно, чтобы два клиента не могли одновременно создать заявки на одну дату и
время, пока первая заявка ещё ждёт обработки админом.

Правило:

```text
pending — слот занят, заявка ждёт решения админа
confirmed — слот занят
rejected — слот свободен
cancelled — слот свободен
```

На уровне БД это дополнительно защищено уникальным busy-slot индексом для
статусов `pending` и `confirmed`.

## 7. Админская логика заявок

В админском меню есть раздел:

```text
Заявки
```

Этот раздел доступен только админу.

Админ определяется по `ADMIN_ID` или списку `ADMIN_IDS` из `.env`.

Лучше перейти от одного `ADMIN_ID` к нескольким:

```env
ADMIN_IDS=123456789,987654321
```


## 10. Подтверждение заявки админом

Когда админ нажимает **Подтвердить**, бот должен:

1. получить заявку по `appointment_id`;
2. проверить, что заявка существует;
3. проверить, что статус заявки `pending`;
4. проверить, что на эту дату и время нет другой занятой заявки;
5. поменять статус на `confirmed`;
6. сохранить изменения;
7. отправить клиенту уведомление.

Сообщение клиенту:

```text
Ваша заявка подтверждена.

Дата: 15.07.2026
Время: 14:00
Эскиз: Череп с цветами

За день до записи бот пришлёт напоминание.
```

Если слот уже занят, бот должен ответить админу:

```text
Этот слот уже занят другой заявкой.
```

## 11. Отклонение заявки админом

Когда админ нажимает **Отклонить**, бот должен:

1. получить заявку по `appointment_id`;
2. проверить, что заявка существует;
3. поменять статус на `rejected`;
4. отправить клиенту уведомление.

Сообщение клиенту:

```text
Ваша заявка отклонена.

Свяжитесь с мастером, чтобы выбрать другую дату или уточнить детали.
```

## 12. Клиентский раздел «Мои заявки»

В клиентском меню есть кнопка:

```text

```

Пользователь должен видеть только свои заявки.

Пример списка:

```text
Ваши заявки:

#8 — 10.07.2026 12:00 — подтверждена
```

При открытии заявки:

```text

Дата: 15.07.2026
Время: 14:00
Статус: ждёт подтверждения
Комментарий: Хочу сделать на предплечье
```

Кнопки:

```text

Назад
```

## 13. Календарь для клиента

Клиентский календарь в главном меню работает как view-only календарь мастера.
Создание заявки начинается через `Запись`.

Он должен:

- показывать дни месяца;
- показывать доступные слоты после выбора доступного дня;
- не показывать количество записей;
- не показывать временные отрезки внутри календаря.

Клиенту не нужно видеть загруженность мастера.

Клиенту важно только:

```text
можно выбрать день или нельзя
```

## 14. Календарь для админа

Админский календарь нужен для контроля записей и выходных.

Он должен показывать:

- дни месяца;
- количество заявок на каждый день;
- пометку выходного дня;
- без временных отрезков на уровне месяца.

Пример кнопок:

```text
1
2
3 · 1
4
5 · 3
6
7 · 2
```

Где:

```text
5 · 3
```

означает:

```text
5 число, 3 заявки
```

## 15. Какие заявки считать в календаре

В счётчик календаря включаем заявки со статусами:

```text
pending
confirmed
```

Не включаем:

```text
rejected
cancelled
```

Причина:

- `pending` важны админу, потому что их надо обработать;
- `confirmed` важны, потому что они реально занимают день;
- `rejected` и `cancelled` уже не актуальны.

## 16. Функция количества заявок по дням

Нужна функция репозитория:

```python
async def count_appointments_by_day(
    session,
    year: int,
    month: int,
) -> dict[date, int]:
    ...
```

Она должна вернуть словарь:

```python
{
    date(2026, 7, 5): 3,
    date(2026, 7, 7): 2,
    date(2026, 7, 15): 1,
}
```

Эта функция используется только для отображения календаря.

## 18. Временный выходной

Временный выходной — это конкретная дата, закрытая один раз.

Пример:

```text
20.07.2026 — выходной
```

Это значит:

- только 20 июля 2026 года недоступно;
- другие такие же дни недели остаются рабочими.

### Сценарий добавления временного выходного

1. Админ открывает календарь.
2. Выбирает дату.
3. Нажимает **Добавить выходной**.
4. Бот спрашивает:

```text
Какой выходной добавить?
```

Кнопки:

```text
Временный
Постоянный
Отмена
```

5. Админ выбирает **Временный**.
6. Бот добавляет временный выходной на конкретную дату.

Сообщение:

```text
Дата 20.07.2026 отмечена как временный выходной.
```

## 19. Постоянный выходной

Постоянный выходной — это выходной по дню недели.

Пример:

```text
Каждое воскресенье — выходной
```

Если админ выбрал дату 19.07.2026, а это воскресенье, и нажал **Постоянный**, бот должен закрыть все воскресенья.

### Сценарий добавления постоянного выходного

1. Админ открывает календарь.
2. Выбирает дату.
3. Нажимает **Добавить выходной**.
4. Бот спрашивает тип выходного.
5. Админ выбирает **Постоянный**.
6. Бот определяет день недели выбранной даты.
7. Бот сохраняет этот день недели как постоянный выходной.

Сообщение:

```text
Воскресенье добавлено как постоянный выходной.
```

## 20. Как хранить постоянные выходные

Лучший вариант — отдельная таблица или отдельная модель.

Например:

```text
WeeklyDayOff
```

Поля:

```text
id
weekday
created_at
updated_at
```

`weekday` — число от 0 до 6.

Логика Python:

```text
0 — понедельник
1 — вторник
2 — среда
3 — четверг
4 — пятница
5 — суббота
6 — воскресенье
```

Не надо создавать отдельную запись на каждое воскресенье.

Храним именно правило.

## 21. Как хранить временные выходные

Временный выходной можно хранить в текущей модели календарных исключений.

Если такой модели ещё нет, добавить:

```text
ScheduleException
```

Поля:

```text
id
date
time
type
created_at
updated_at
```

Для временного выходного:

```text
type = temporary_day_off
date = 2026-07-20
time = null
```

Для заблокированного слота:

```text
type = blocked_slot
date = 2026-07-20
time = 14:00
```

## 22. Снятие выходного

Админ должен иметь возможность снять выходной.

### Если дата временно выходная

Показываем кнопку:

```text
Снять временный выходной
```

После нажатия удаляем временное исключение.

### Если день недели постоянный выходной

Показываем кнопку:

```text
Снять постоянный выходной
```

После нажатия удаляем правило из `WeeklyDayOff`.

### Если есть оба типа

Например, дата попадает под постоянный выходной и ещё отдельно отмечена как временный выходной.

Тогда показываем:

```text
Статус: выходной

Причины:
- постоянный выходной: воскресенье
- временный выходной: 20.07.2026
```

И две кнопки:

```text
Снять временный выходной
Снять постоянный выходной
```

## 23. Блокировка отдельного слота

Блокировка слота — это закрытие конкретного времени в конкретный день.

Пример:

```text
15.07.2026

10:00 — свободно
12:00 — свободно
14:00 — заблокировано
16:00 — свободно
```

Блокировка нужна, если мастер занят в конкретное время, но весь день закрывать не нужно.

В календаре месяца эти слоты не показываем.

Они показываются только после открытия конкретного дня.

## 24. Доступность даты

Дата недоступна клиенту, если:

1. дата в прошлом;
2. дата отмечена как временный выходной;
3. день недели является постоянным выходным;
4. на эту дату нет ни одного свободного слота.

Дата доступна, если:

1. дата сегодня или в будущем;
2. это не временный выходной;
3. это не постоянный выходной;
4. есть хотя бы один свободный слот.

## 25. Доступность слота

Слот недоступен, если:

1. дата недоступна;
2. слот заблокирован вручную;
3. на этот слот уже есть заявка со статусом `pending` или `confirmed`.

`pending`-заявки блокируют слот до подтверждения, отклонения или отмены.

## 26. Напоминания клиентам

Реализовано напоминание за день до записи.

Условие отправки:

```text
status = confirmed
appointment_date = tomorrow
reminder_sent = false
```

Время отправки:

```text
12:00 за день до записи
```

Текст:

```text
Напоминание о записи

Завтра у вас сеанс тату.

Дата: 15.07.2026
Время: 14:00
Эскиз: Череп с цветами

Если планы изменились — напишите мастеру заранее.
```

После отправки:

```text
reminder_sent = true
```

## 27. APScheduler

Для напоминаний использовать APScheduler.

Минимальный вариант:

```text
каждый день в 12:00 запускать send_tomorrow_reminders()
```

Функция должна:

1. найти подтверждённые заявки на завтра;
2. отфильтровать заявки, где `reminder_sent = false`;
3. отправить сообщение пользователю;
4. обновить `reminder_sent = true`.

## 28. Модели

Фактические поля нужно сверять с `db.models`. Функционально используются такие
сущности.

### User

Пользователь Telegram.

Хранит:

```text
id
telegram_id
username
created_at
updated_at
```

Опционально:

```text
full_name
role
```

Если роль не хранится в БД, админа можно определять через `ADMIN_IDS`.

### Style

Стиль эскиза.

Хранит:

```text
id
name
created_at
updated_at
```

### Sketch

Эскиз.

Хранит:

```text
id
style_id
name
description
price
photo_file_id
status
views
created_at
updated_at
```

### Appointment

Заявка.

Хранит:

```text
id
user_id
sketch_id | null
appointment_date
appointment_time
request_type
client_sketch_photo_file_id | null
status
comment
reminder_sent
created_at
updated_at
```

`request_type` принимает значения:

```text
catalog_sketch
custom_sketch
no_sketch
```

### ScheduleException

Исключение календаря.

Хранит:

```text
id
date
time
type
created_at
updated_at
```

Типы:

```text
temporary_day_off
blocked_slot
```

### WeeklyDayOff

Постоянный выходной.

Хранит:

```text
id
weekday
created_at
updated_at
```

## 33. Deployment

`main` используется для Railway/webhook deployment:

- `BOT_MODE=webhook` по умолчанию;
- `WEBHOOK_URL`, `WEBHOOK_PATH`, `WEBHOOK_SECRET_TOKEN`,
  `WEBHOOK_HOST`, `WEBHOOK_PORT` остаются актуальными;
- Tailscale/Funnel service, volume и env-переменные отсутствуют;
- миграции применяются внутри `bot.main` перед стартом webhook/polling.

Docker Compose в `main` содержит только `bot` и `redis`. Если нужен deployment
через Tailscale Funnel, использовать ветку `tailscale-funnel`.

## 34. Критерии готовности

Функционал готов, если:

- пользователь может выбрать эскиз;
- пользователь может открыть `Запись`;
- пользователь может создать заявку из каталожного эскиза;
- пользователь может создать заявку с фото своего эскиза;
- пользователь может создать заявку без эскиза;
- пользователь может выбрать дату;
- пользователь может выбрать время;
- пользователь может оставить комментарий;
- заявка создаётся со статусом `pending`;
- пользователь видит свои заявки;
- админ видит список заявок;
- админ может подтвердить заявку;
- админ может отклонить заявку;
- клиент получает сообщение о подтверждении;
- клиент получает сообщение об отклонении;
- подтверждённая заявка занимает слот;
- pending-заявка занимает слот до обработки;
- админский календарь показывает количество заявок на день;
- календарь месяца не показывает временные отрезки;
- админ может добавить временный выходной;
- временный выходной закрывает только одну дату;
- админ может добавить постоянный выходной;
- постоянный выходной закрывает все выбранные дни недели;
- клиент не может выбрать выходной день;
- клиент не может выбрать заблокированный слот;
- за день до подтверждённой записи клиент получает напоминание.
