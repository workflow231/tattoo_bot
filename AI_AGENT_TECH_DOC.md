- `tailscale-funnel` — отдельная ветка с Docker Compose deployment через
  Tailscale Funnel.

## 1. Главная цель задачи

Поддерживать и развивать текущую логику записи без регрессий:

- клиент начинает через `Запись`;
- `Выбрать эскиз` открывает старый каталог;
- `Мой эскиз` принимает только Telegram photo;
- `Заявка без эскиза` создаёт заявку с пустым `sketch_id`;
- админ видит, подтверждает и отклоняет заявки;
- pending+confirmed блокируют слот;
- Railway/main deployment остаётся без Tailscale wiring.

## 3. Ключевой сценарий

### Текущий поток записи

```text
Запись
→ Мой эскиз | Выбрать эскиз | Заявка без эскиза | Чат с мастером
→ выбрать дату
→ выбрать время
→ оставить комментарий или пропустить
→ подтвердить создание
→ заявка создаётся со статусом pending
```

`Выбрать эскиз` открывает каталог:

```text
выбор стиля
→ выбор эскиза
→ карточка эскиза
→ Создать заявку
```

`Мой эскиз` сохраняет `client_sketch_photo_file_id`; админ должен видеть фото в
карточке заявки. `file_id` не логировать.

## 4. Модель Appointment

Модель заявки уже есть. Ключевые поля:

```python
id: int
user_id: int
sketch_id: int | None
appointment_date: date
appointment_time: time
request_type: str
client_sketch_photo_file_id: str | None
status: str
comment: str | None
reminder_sent: bool
created_at: datetime
updated_at: datetime
```

Статусы:

```text
pending
confirmed
rejected
cancelled
```

Правила:

- новая заявка всегда создаётся со статусом `pending`;
- `pending` занимает слот до обработки админом;
- `confirmed` занимает слот;
- `rejected` не занимает слот;
- `cancelled` не занимает слот.
- `request_type` принимает `catalog_sketch`, `custom_sketch`, `no_sketch`.
- `sketch_id` обязателен только для `catalog_sketch`.
- `client_sketch_photo_file_id` обязателен только для `custom_sketch`.

## 7. AppointmentService

Сервис заявок держит бизнес-валидации. Ключевые методы:

```python
create_pending_appointment()
confirm_appointment()
reject_appointment()
```

### create_pending_appointment

Должен:

1. проверить пользователя;
2. проверить `request_type`;
3. для `catalog_sketch` проверить эскиз;
4. для `custom_sketch` проверить наличие фото;
5. проверить, что слот не занят `pending` или `confirmed`;
6. создать заявку со статусом `pending`;
7. сохранить комментарий, если он есть.

### confirm_appointment

Должен:

1. получить заявку;
2. проверить, что статус `pending`;
3. проверить, что нет другой занятой заявки на этот же день и время;
4. поменять статус на `confirmed`;
5. вернуть результат для хендлера.

### reject_appointment

Должен:

1. получить заявку;
2. поменять статус на `rejected`;
3. вернуть результат для хендлера.

## 8. Клиентский handler appointments.py

Клиентский handler заявок должен сохранять текущую навигацию и FSM.

Он должен обрабатывать:

```text
Запись
Мой эскиз

Заявка без эскиза
выбор даты
выбор времени
комментарий
подтверждение создания заявки

```

Состояния:

```python
AppointmentState.choosing_date

```

Важные правила:

- `Запись` не открывает каталог напрямую;
- `Выбрать эскиз` открывает старый каталог;
- кнопка `Создать заявку` работает из карточки эскиза;
- если в FSM нет `sketch_id`, ответить: `Сначала выберите эскиз из каталога.`
- в `custom_sketch` принимать только Telegram photo;
- после stale FSM пользователь не должен попадать в тупик.

## 12. Календарь

Календарь нужен в двух режимах:

1. клиентский;
2. админский.

### Клиентский календарь

Клиентский календарь показывает:

- дни месяца;
- доступные/недоступные дни;
- без количества заявок;
- без временных слотов на уровне месяца.

### Админский календарь

Админский календарь показывает:

- дни месяца;
- количество заявок на день;
- отметку выходного;
- без временных слотов на уровне месяца.

Пример:

```text
1
2
3 · 1
4
5 · 3
6
7 · 2
```

## 15. Временные выходные

Временный выходной — конкретная дата, закрытая один раз.

Пример:

```text
20.07.2026 — выходной
```

Требование:

- закрывает только одну дату;
- другие такие же дни недели остаются рабочими.

Хранить как календарное исключение:

```text
type = temporary_day_off
date = 2026-07-20
time = null
```

## 16. Постоянные выходные

Постоянный выходной — выходной по дню недели.

Пример:

```text
каждое воскресенье — выходной
```

Лучше хранить отдельной таблицей/моделью:

```text
WeeklyDayOff
```

Поля:

```text
id
weekday
created_at
updated_at
```

`weekday`:

```text
0 — понедельник
1 — вторник
2 — среда
3 — четверг
4 — пятница
5 — суббота
6 — воскресенье
```

Не создавать отдельные записи на каждую будущую дату.

## 17. Снятие выходных

Нужно сделать:

- снять временный выходной;
- снять постоянный выходной.

Если дата одновременно попадает под постоянный и временный выходной, показывать обе причины.

Пример:

```text
Статус: выходной

Причины:
- постоянный выходной: воскресенье
- временный выходной: 20.07.2026
```

## 18. Блокировка слота

Блокировка слота — закрытие конкретного времени в конкретный день.

Пример:

```text
15.07.2026

10:00 — свободно
12:00 — свободно
14:00 — заблокировано
16:00 — свободно
```

Хранить как календарное исключение:

```text
type = blocked_slot
date = 2026-07-15
time = 14:00
```

В месячном календаре слоты не показывать.

## 19. Логика доступности даты

Дата недоступна клиенту, если:

1. дата в прошлом;
2. дата является временным выходным;
3. день недели является постоянным выходным;
4. на эту дату нет ни одного свободного слота.

Дата доступна, если:

1. дата сегодня или в будущем;
2. это не временный выходной;
3. это не постоянный выходной;
4. есть хотя бы один свободный слот.

## 20. Логика доступности слота

Слот недоступен, если:

1. дата недоступна;
2. слот заблокирован вручную;
3. на этот слот уже есть заявка со статусом `pending` или `confirmed`.

`pending`-заявки блокируют слот до подтверждения, отклонения или отмены.

### How to verify

Команды проверки и ручной сценарий:

```text
1. Запустить бота.
2. Открыть /start.
3. Открыть `Запись`.
4. Проверить `Выбрать эскиз`: стиль → эскиз → `Создать заявку`.
5. Проверить `Мой эскиз`: принимает фото, не принимает текст.
6. Проверить `Заявка без эскиза`.
7. Выбрать дату и время.
8. Создать заявку.
9. Вторым пользователем попробовать тот же слот: слот должен быть недоступен.
10. Зайти под админом.
11. Открыть `Заявки`.
12. Подтвердить или отклонить заявку.
13. Проверить уведомление клиенту.
```

### Risks / edge cases

Указать риски:

- пустой каталог;
- нет доступных слотов;
- заявка уже подтверждена;
- слот уже занят;
- пользователь удалил чат с ботом;
- reminder_sent не обновился;
- timezone;
- старая FSM-сессия;
- админ нажал кнопку по старой заявке.
